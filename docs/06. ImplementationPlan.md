## 3. Step-by-Step Build

### Step 1 вЂ” Fetch & Cache TLE Data (`backend/fetch_tle.py`)
Implementation per `TechSpec.md В§4.1`. Key behaviors:
- Fetch a dense same-constellation subset (Starlink recommended вЂ” same-shell objects pass near each other more often, which materially de-risks "zero alerts found" during the demo).
- Fetch 2вЂ“5 named Indian satellites individually by NORAD ID (Cartosat, RISAT, EOS, IRNSS series вЂ” **confirm exact current NORAD IDs against Celestrak at build time**, do not hardcode from memory, satellite catalogs change).
- Cache both the TLE text and a metadata JSON (fetch timestamp, count).
- **Re-run this fetch immediately before any recorded demo** вЂ” TLE accuracy degrades with time since epoch, so "fresh right before demo" is a hard operational rule, not a suggestion.

### Step 2 вЂ” Propagate Orbits (`backend/propagate.py`)
Implementation per `TechSpec.md В§4.2`.
**Sanity check before moving to Step 3:** print a few computed position magnitudes and confirm they fall in the 6,700вЂ“8,000 km range (Earth radius ~6,371 km + LEO altitude). If numbers look wildly off, the TLE parsing or timescale is wrong вЂ” fix this before building anything on top. This check should also be encoded as an automated assertion (see `TechSpec.md В§8`), not just eyeballed once.

### Step 3 вЂ” Conjunction Screening (`backend/screen.py`)
Implementation per `TechSpec.md В§4.3`: pairwise `itertools.combinations` + vectorized `numpy` distance, threshold default 50km (clearly labeled a monitoring threshold, per `PRD.md` FR-3.2), plus the fine-refinement pass (`TechSpec.md В§4.2`) for any flagged pair.

**If you get zero alerts:** first widen `threshold_km`; second, confirm the constellation group is a single dense constellation (Starlink/OneWeb), not a mixed "active" set. This troubleshooting order is important вЂ” don't start debugging code before trying the parameter/dataset fix, which is usually the actual issue.

### Step 4 вЂ” Risk Scoring (`backend/risk_score.py`)
Implementation per `TechSpec.md В§4.4`: two-factor (proximity 70% + closing-speed 30%) deterministic formula, LOW/MEDIUM/HIGH bucketing at the same 35/70 cut points as the original draft (kept for consistency), monitoring window scaled by risk level.

### Step 5 вЂ” API Layer (`backend/api.py`)
Wire the above three modules behind the FastAPI routes specified in `TechSpec.md В§5`. Even for the Round 1 minimal build, stand up at minimum `GET /api/satellites` and `POST /api/screen` вЂ” the frontend needs a real API to call rather than mocked data, and this is a small amount of extra work for a much more credible "working prototype" claim.

```bash
uvicorn api:app --reload --port 8000
```

### Step 6 вЂ” 3D Visualization (`frontend/src/scene/OrbitScene.tsx`)
This is the highest-leverage, highest-risk step вЂ” start it in **Week 1**, not last, and treat it as a spike:
1. Get a `<Canvas>` rendering a sphere for Earth and `OrbitControls` (from `@react-three/drei`) for free-rotate/zoom.
2. Render one satellite's orbit as a `<Line>` from a hardcoded/mocked position array вЂ” validate the rendering approach before wiring real API data.
3. Wire real position data from `POST /api/screen`.
4. Add the alert highlight treatment (glow, pulse, connecting line) per `Design.md В§4`.
5. Add the camera fly-to-alert animation (`Design.md В§5`) вЂ” this is the actual "winning moment," budget real time for getting the easing/timing right, don't leave it for the last hour.

**Decision checkpoint:** if this isn't rendering orbits + fly-to by the Week-2 checkpoint in `Tracker.md`, switch to the documented Plan B (Streamlit + `pydeck`) per `TechSpec.md В§2.3` rather than risk arriving at the deadline with neither a finished 3D scene nor a finished fallback.

### Step 7 вЂ” Dashboard Shell (`frontend/src/dashboard/*`)
Build `ControlPanel`, `AlertsPanel`, `AlertCard`, `RiskBadge`, `ScoreBreakdown`, `PerceiveReasonExecuteStrip`, `DisclaimerBar` per the component specs in `Design.md В§6`. Wire to the Zustand store (`TechSpec.md В§6`) so selecting an alert card and clicking directly in the 3D scene both drive the same `selectedAlertId` state.

### Step 8 вЂ” Integration & Polish
- Full frontendв†”backend integration test: run the whole flow end to end with a fresh TLE fetch.
- Error/fallback states per `AppFlow.md В§2.7` вЂ” test by deliberately breaking network access and confirming the cached-data warning banner appears correctly instead of a crash.
- Timing pass on animations per `Design.md В§5`.
- Write the mandatory `README.md` (project description, setup instructions, architecture summary linking to `TechSpec.md`, disclaimer section, license note).
- Add `LICENSE` (MIT, per `Rules.md`).

### Step 9 вЂ” Demo Recording & Deck Finalization
- Fresh TLE re-fetch immediately before recording.
- Record the demo video per the script in `AppFlow.md В§5` (в‰¤3вЂ“4 minutes, per `Rules.md`).
- Finalize the 7-point deck per `PRD.md В§16`.
- Take backup screenshots/GIFs in case of any last-minute recording issues.

## 6. Known Risks & Mitigations (see `PRD.md В§13` for the full table; build-specific subset below)

| Risk | Mitigation |
|---|---|
| Zero conjunctions found in the working dataset | Use same-constellation TLEs (Starlink/OneWeb), widen threshold to 50вЂ“100km, clearly labeled as a monitoring threshold |
| 3D scene laggy with too many objects/timesteps | Cap to 15вЂ“20 satellites, 12вЂ“24h window; profile early in Phase 2, not the night before |
| TLE accuracy degrades over propagation time | Fetch fresh TLEs right before any demo recording; keep propagation window short |
| React/Three.js build stalls | Named fallback (`TechSpec.md В§2.3`) with a hard decision checkpoint at the Week-2 mark |
| Live demo breaks (wifi/API down) during actual judging | Cache all TLE data locally in advance; always have a pre-recorded backup video |
| Team re-derives the "single overnight hackathon" mistake for Round 2 | Explicitly re-read `PRD.md В§3` and `Rules.md` at the start of Round 2 planning |
